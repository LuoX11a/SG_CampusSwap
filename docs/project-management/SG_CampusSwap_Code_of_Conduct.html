📜 3. Code of Conduct & Expectations
  • 1 Respect & Inclusivity — All team members will treat each other with respect. We value diverse opinions and foster an inclusive environment. No form of harassment, discrimination, or disrespectful language will be tolerated.
  • 2 Accountability — Each member takes ownership of their assigned tasks. If a deadline cannot be met, the member must notify the Project Manager at least 24 hours in advance to arrange support or redistribution.
  • 3 Punctuality — Attend all scheduled meetings on time. If unable to attend, notify the team at least 2 hours before the meeting and catch up via meeting minutes.
  • 4 Quality & Integrity — All code must be reviewed before merging. No plagiarism. All external sources must be properly cited. Commit messages must be clear and descriptive.
  • 5 Constructive Feedback — Give feedback that is specific, actionable, and kind. Receive feedback with an open mind. The goal is team growth, not blame.
  • 6 Transparency — Share progress, blockers, and concerns openly. No hiding problems — early communication prevents last-minute crises.
  • 7 Contribution Balance — Workload should be distributed fairly. If any member feels overburdened or underutilized, they should raise it with the PM immediately.
  • 8 Continuous Improvement — Each sprint retrospective must produce at least one actionable improvement item. The team commits to implementing it in the next sprint.
⚖ 4. Decision-Making & Conflict Resolution

📊 Decision-Making Process

  • 1 Consensus First — The team aims for consensus on all major decisions (scope changes, architecture choices, timeline adjustments).
  • 2 Majority Vote — If consensus cannot be reached after thorough discussion, a majority vote (4 of 6) decides. The PM casts the tie-breaking vote if needed.
  • 3 PM Authority — For urgent operational decisions (e.g., merging a hotfix, reassigning a task due to illness), the PM has authority to decide unilaterally and inform the team afterward.
  • 4 Scope Changes — Any feature addition after Week 4 requires a team vote and timeline re-evaluation. The default answer is "add to Future Version backlog."

💻 5. Technical Work Practices
  • 1 Branching Strategy — main branch is always deployable. Feature branches named feature/<name>. Bug fixes in fix/<name>. Never commit directly to main.
  • 2 Code Review — Every PR requires at least one approving review from a different team member. Review within 24 hours of request.
  • 3 Documentation — API endpoints documented in the shared Google Doc. README kept up-to-date. Complex logic should have inline comments explaining why, not what.
  • 4 Testing — Write unit tests for critical paths (auth, listing CRUD, chat). Manual testing checklist before each sprint review. Bugs logged in GitHub Issues with clear reproduction steps.
  • 5 Commits — Commit messages follow conventional commits: feat:, fix:, docs:, refactor:, test:. Commit early, commit often.
✍ 6. Team Commitment Statement

As members of the SG CampusSwap project team, we collectively agree to:

✓ Dedicate approximately 6–8 hours per week to this project outside of scheduled meetings.
✓ Complete all assigned tasks by agreed-upon deadlines, or communicate early if unable to do so.

✓ Uphold the principles of respect, transparency, and accountability in all team interactions.
✓ Produce our best work — this project represents our collective capability to future employers and stakeholders.
✓ Support fellow team members when they face challenges. We succeed or fail as a team.
